# Codex 模型目录可靠发现设计

## 目标

Quotio 在不同机器和不同 Codex 安装形态下，都能自动找到实际可用的 Codex CLI 二进制、提取模型目录并生成 `~/.codex/quotio-model-catalog.json`，不再要求用户从其他电脑手工复制文件。

本次修复覆盖 WindowsApps 桌面版、系统 `PATH`、Codex 本地缓存目录和 `~/.codex/bin`。Codex 上游未来若改变内嵌目录格式，Quotio 仍需保持界面响应，并给出明确失败状态，而不是无提示地永久回退到四档。

## 根因

当前 `locate_codex_cli()` 只检查 `%LOCALAPPDATA%/OpenAI/Codex/bin`、平台应用数据目录和 `~/.codex/bin`。新安装的 WindowsApps 机器可能只有已探测到的 `.../app/Codex.exe`，真正包含模型目录的 CLI 位于其同级 `resources/codex.exe`；本地 bin 缓存尚未生成时，定位函数直接返回 `None`。

现有定位还有第二个缺陷：根目录存在旧版 `codex.exe` 时会立即返回，不再比较哈希子目录或系统 `PATH` 中的新版，因此可能用旧目录覆盖新版 Codex 的模型目录。

## 方案选择

采用“多来源候选 + 实际内容验证 + 上一份可用目录兜底”的方案。

不采用只补一条 WindowsApps 路径的最小补丁，因为安装布局再次变化时会复发；也不内置固定 JSON 快照，因为 `model_catalog_json` 可能替换 Codex 自带目录，过期快照会隐藏新模型。

## 二进制候选发现

候选来源如下：

1. `codex_launch::detect_codex_app_path_cached()` 已探测到的桌面应用：
   - Windows：从 `.../app/Codex.exe` 派生 `.../app/resources/codex.exe`。
   - macOS：检查应用包内与主程序相邻的常见资源位置。
2. 系统 `PATH` 中可执行的 `codex` / `codex.exe`。
3. `%LOCALAPPDATA%/OpenAI/Codex/bin`、macOS/Linux 对应应用数据目录及其一级哈希子目录。
4. `~/.codex/bin` 及其一级子目录。

所有路径按规范化结果去重。候选按修改时间从新到旧排列；修改时间相同则保持来源优先级。不会因为根目录存在一个文件就提前结束候选收集。

## 提取与缓存流程

`ensure_catalog()` 的流程调整为：

1. 获取并去重候选列表。
2. 若现有元数据指向某个候选，且其指纹未变、目录 JSON 仍通过结构校验，直接返回缓存。
3. 否则按顺序逐个候选运行真实目录提取；某个候选打不开、没有锚点或 JSON 校验失败时继续下一个。
4. 第一份成功提取的目录通过临时文件加重命名的方式原子写入，随后更新来源指纹元数据。
5. 所有候选均失败时，如果磁盘上已有一份结构合法的目录，则作为“上一份可用目录”继续使用，但不伪造新的来源指纹。
6. 没有候选且没有合法旧目录时返回可诊断失败；前端可暂时显示保守档位，但不能把它表现成已成功读取模型目录。

进程内增加单次执行保护，页面读取档位和启动方案同时触发时，只允许一轮二进制扫描，其余调用复用结果。

## 界面与性能

`fetch_codex_reasoning_levels` 改为异步 Tauri 命令，文件扫描通过 `spawn_blocking` 在后台线程执行。首次扫描可能耗时约 0.5 至 2 秒，但不占用 UI 线程。

方案编辑器在等待时保持当前选择可见，并显示加载状态；成功后刷新档位。缓存命中时只读取小型元数据和目录 JSON，不重新扫描数百 MB 二进制。

启动方案已有后台阻塞任务，生成配置时可安全等待同一份扫描结果，不冻结窗口。

## 错误处理

- 单个候选失败：继续尝试其他候选。
- 新目录写入失败：保留旧目录，不留下半写文件。
- 元数据写入失败：目录仍可使用，下次重新验证来源。
- 上游格式变化：使用合法的上一份目录，并记录可诊断原因。
- 完全没有可用目录：返回明确状态，前端回退但不静默声称读取成功。

## 测试

新增回归测试覆盖：

1. 没有本地 bin，仅存在 `app/resources/codex.exe` 时能够发现候选。
2. 根目录旧版与哈希子目录新版并存时选择新版，不被旧文件遮挡。
3. 第一个候选不含合法目录时继续尝试第二个候选。
4. 所有候选失败但旧目录合法时保留上一份可用目录。
5. 原子写入失败不会破坏已有目录。
6. 异步 Tauri 命令仍返回对应模型的动态档位。

完成后运行目标单测、`cargo test --workspace`、前端测试、前端生产构建和格式检查。
