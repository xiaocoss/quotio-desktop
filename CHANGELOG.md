## v0.7.1 - 2026-07-19

### 新增

- **Codex 启动方案可选 Dream Skin**：Windows 商店版的应用启动方案可显式开启主题注入；Quotio 统一负责启动、验证、停止、切换与异常清理，并随安装包和便携版携带运行资源。默认关闭，运行时需要 Node.js 22 或更新版本。
- **Dream Skin 内置主题选择**：启动方案可选择「梦境粉紫」「极光青蓝」「午夜星河」，停止方案后编辑并重新启动即可切换。
- **Dream Skin 自定义主题库**：可从启动方案直接导入 PNG、JPEG 或 WebP 图片，Quotio 会生成并保存独立用户主题，之后可在主题下拉中重复选择。
- **内置 README 八套效果主题**：新增「粉系定制、财神打工版、红白科幻、清透定制、灵感小宇宙、紫夜限定、初音未来、舞台黑金」，随免安装包一同分发；其中人物/IP 图仅限个人使用。

### 修复

- **修复免安装版 Dream Skin 无法启动**：Tauri 资源路径带 `\\?\` 前缀时，Windows PowerShell 5.1 会令 `$PSScriptRoot` 为空；启动器现在会转换为 PowerShell 兼容路径，并统一使用 UTF-8 输出错误信息。

## v0.6.4 - 2026-07-15

修复商店版 Codex 在部分系统(尤其 Windows LTSC 精简版)上一键启动"起来一下就退、根本用不成"的问题。

### 修复

- **商店版(MSIX)Codex 一键启动改为一律走 `shell:AppsFolder` 激活,不再直接 spawn / 按 exe 路径拉起**。此前启动是多层兜底:先直接 `spawn` exe(被 ACL 拒才走 shell)、shell 里又先试 exe 路径 `Start-Process`(失败才用商店入口)。问题在于:部分系统(实测 **Windows LTSC**)上,直接按 exe 路径拉起 MSIX 会起一个**不完整的实例、随即自退**,而 `spawn`/`Start-Process` 却"假成功"返回 —— Quotio 于是以为启动成功、不再走正确的 `shell:AppsFolder\<AppUserModelId>` 激活,结果 Codex 永远"闪一下就没"。现在 WindowsApps 路径直接用商店入口激活(用户在 LTSC 上实测该方式能干净启动),`spawn`/exe 路径仅作非 MSIX / 绿色版的兜底。承接 0.6.2(进程名)、0.6.3(监控)一起,彻底解决 LTSC 上"Codex 启动不了"。

## v0.6.3 - 2026-07-15

进一步加固 Codex 一键启动的稳健性(承接 0.6.2)。

### 修复

- **Codex 一键启动后不再因"探测不到进程"就被误判失败、自动还原配置**。此前监控若在整个会话里从未探测到 Codex 进程(典型:商店版进程名叫 `ChatGPT.exe`、或 MSIX shell 激活拿不到子进程句柄),会在 60 秒后当作"启动失败",把 Codex 配置还原并清掉会话 —— 结果把明明已经起来的 Codex 判死、配置改回,用户就用不成了(表现为"Codex 起来一下又没了 / 一键启动没反应",且退了杀毒软件也无效)。现在改为:**只有先明确观测到 Codex 在跑、之后又连续丢失(即用户手动退出)才自动还原;从头到尾没观测到的,一律信任其已启动、绝不激进关闭**。既然启动前已探测到应用路径、也发了启动命令,探测不到进程更可能是"检测方法"的问题而非真没起来。配合 0.6.2 的进程名匹配扩充(`codex*` + `ChatGPT.exe`),双重保险,即便进程名再变也不会误伤。

## v0.6.2 - 2026-07-12

修复 Windows 商店版 Codex 关不掉 / 反复启动残留累积、导致「启动不起来」的问题。

### 修复

- **修复 Windows 商店版 Codex(进程名 `ChatGPT.exe`)关不掉、检测不到、残留累积到「启动不起来」**。Quotio 的「关闭旧实例 / 检测运行 / 启动后 pid 探测」三处都只按 `codex*` 匹配进程名,而 OpenAI 把商店版 Codex 的进程名叫 `ChatGPT.exe`(与 macOS 上外壳 `Codex.app`、主程序却叫 `ChatGPT` 是同源的交叉命名),`codex*` 根本匹配不到 —— 于是每次「启动前先关旧实例」形同虚设,旧 `ChatGPT.exe` 越攒越多,商店版是单实例、被一堆残留占着,新窗口激活失败,表现就是点一键启动没反应。现在三处都同时匹配 `codex*` 与 `ChatGPT.exe`,启动前能真正清掉残留、干净启动。

## v0.6.0 - 2026-07-12

这一版是一次大规模界面重构:全部主界面按设计稿重做、托盘悬浮窗重写,并新增一个让「关掉 Quotio 不再连累 Codex」的实用开关。

### 优化

- **托盘悬浮窗照设计稿重做**:窄长面板 + 顶部(服务状态 / 本地地址 / 启停按钮)、可换行的智能体筛选、可滚动的账号额度卡(邮箱 / 健康度 / PLUS / 到期 / 倒计时 / 健康点阵 / Session·Weekly 进度条),底部固定的刷新 / 打开 / 退出三行操作。
- **悬浮窗改为常驻置顶**:点到别的应用不再自动隐藏,避免「悬浮窗莫名消失」;通过再次点击托盘图标或「打开 Quotio」收起。
- **全部主界面重设计**(仪表盘 / 额度 / 服务商 / 2FA / 智能体 / 日志 / 设置 / 关于):统一冷白背景、纯白卡片、克制的边框与淡阴影;去掉了会拖慢 WebView 的噪点纹理与重阴影。
- **额度页与服务商页改用各家品牌 logo**(Codex→OpenAI、Claude→Anthropic、Copilot→GitHub、Antigravity→Gemini、DeepSeek 等),白圆底 + 品牌色。

### 修复

- 悬浮窗前端注释与后端行为不一致(注释称不随失焦隐藏、后端实际会隐藏)——现已统一为常驻,不再随失焦消失。
- 一批重设计期间的交互问题:2FA 未输入密钥时倒计时误走、设置开关圆点漂移出控件、服务商自定义接口展开后按钮文案不变、日志「查看详情」无法展开等。

## v0.5.13 - 2026-07-11

这一版继续修 macOS 上的 Codex 探测。上一版兼容了 `ChatGPT.app`，但实机上还有一种「外壳名和主程序名对不上」的组合没覆盖到。

### 修复

## v0.5.12 - 2026-07-11

这一版是配置保护补丁。重点修复升级或启动后 Codex 启动方案被空列表覆盖，导致方案看起来“消失”的问题。

### 修复

- **已有 `settings.json` 时不再整份用空状态覆盖**。普通设置保存会合并更新已有配置属性，并保留未知字段，避免某次前端/启动迁移拿到的 stale 空状态把配置文件写坏。
- **Codex 启动方案增加防清空保护**。磁盘里已有 `codex_profiles` 且本次保存传入空数组时，默认保留原方案；只有用户在界面里明确删除最后一个方案时，才允许把方案列表清空。
- **旧方案缺少新增字段时不再整组丢失**。例如升级前的方案没有 `proxy_url` 等新字段时，会按默认值补齐并保留方案，而不是解析失败后回退成空列表。
- **写入配置前会自动备份**。后续即使再遇到异常写入，也能从备份里恢复最近一次配置。
- **内存状态同步实际落盘结果**。如果保存时保护了磁盘里的方案，返回给界面的状态也会带回这些方案，避免文件保住了但界面仍显示为空。

## v0.5.11 - 2026-07-11

这一版继续修 Codex 启动方案。重点是：换机器、新装 Codex、或 macOS 只装了 `ChatGPT.app` 时，Quotio 不再卡在旧的探测路径上。

### 修复

- **macOS 探测不到 Codex 桌面应用已修复**。新版安装里桌面端可能叫 `ChatGPT.app`，而旧逻辑只找 `/Applications/Codex.app`，导致界面提示「未探测到 Codex 应用」。现在同时兼容 `Codex.app` 和 `ChatGPT.app`，包括应用路径探测、启动后的进程检测和关闭旧实例。
- **换机器后读取不到 Codex 模型目录、思考程度无法更新的问题进一步加固**。Quotio 现在会从更多真实安装位置查找 Codex 原生二进制，包括 WindowsApps、npm 全局/嵌套安装、macOS app bundle 资源目录、Linux XDG 数据目录和 `PATH`，不会只依赖某一台机器上碰巧存在的缓存文件。
- **生成 `quotio-model-catalog.json` 更稳了**。目录提取结果会先校验结构，再用原子写入替换；如果新候选二进制提取失败，会保留上一份可用目录，避免把配置写坏后必须手动从别的电脑拷贝覆盖。
- **启动方案写 Codex 配置时增加串行保护**。同一时间多处刷新、启动、修复可见性不会并发改同一批 Codex 配置文件，降低配置被互相覆盖或写一半的风险。

### 改进

- **读取 Codex 思考档位不再阻塞界面**。模型目录提取和配置刷新改为后台执行；界面先保持可操作，读取完成后再更新模型/思考程度选项。

## v0.5.10 - 2026-07-10

用启动方案跑 Codex 的,这一版把**丢失的推理档位**找回来了。

### 修复

- **用启动方案启动 Codex 时,推理强度只剩四档(丢了「超高」和「最高」)已修复**。Codex 的可用推理档位是按模型查表得来的 —— 官方登录时这张表由服务端下发,而启动方案会把 Codex 指向本地代理,于是拿不到表,只能退回通用的四档。现在 Quotio 会自动从你本机安装的 Codex 里读出这张表并交给它,`GPT-5.6 Sol` 这类模型的六档就都回来了。这张表**跟着你的 Codex 一起更新**(Codex 一升级就自动重新读取),不是写死的。
- **启动方案不再把你在 Codex 里选好的推理强度打回去**。此前方案里最高只能选到「极高」,一启动就会把 Codex 里选的「最高」覆盖掉。现在方案的「思考程度」按所选模型动态给出可选档位:`GPT-5.6 Sol` 给六档,`GPT-5.5` 给四档;以后 Codex 新增档位也不用等 Quotio 更新。
- **「思考程度」里那个选了没用的「最低」已移除**。Codex 的模型里没有任何一个支持这一档,选中它只会往配置里写一个无效值。
- **档位名称纠正**:上一版把「极高」误改成了「超高」。现在是 低 → 中 → 高 → **极高** → **超高** → **最高**。(Codex 自己把其中两档都显示成「极高」,我们给中间那档换了个名字以便区分。)日志页、设置页、启动方案三处现在用同一套名称。

## v0.5.9 - 2026-07-10

一版修复。**如果你在受限地区、重新授权 Codex 时卡在「Token 交换失败(403)」,这一版必装。**

### 修复

- **重新授权 Codex 时报 `Token 交换失败(403) unsupported_country_region_territory` 已修复**。浏览器里的授权那一步能过,是因为它走了你系统的代理;但 Quotio 自己发出的「用授权码换 token」那一次请求此前是**直连**的,上游按 IP 判定地区不支持,直接拒绝。现在这个请求会走你在设置里填的「代理地址」(没填则回退读系统的代理环境变量),并且支持 `socks5://` 形式的代理。**前提是你在设置里填了代理地址**,否则它仍然直连。
- **下拉菜单被下方的卡片盖住半截已修复**。仪表盘的筛选下拉(以及其它面板里的下拉)展开后,底部选项会被下面的统计卡片压住、点不到。现在下拉菜单会正确地浮在最上层,并且滚动页面时始终跟着按钮走。
- **改「禁用图像生成」开关后不再需要手动重启代理**。此前改完这个开关代理并不会真正应用新设置,表现为「改了跟没改一样」——生图一直 404,或者 Codex 一直报工具冲突。现在改动它会自动重启代理让设置立即生效。

### 改进

- **「思考程度」的档位名称与 Codex 保持一致**:`超高`(此前叫「极高」,而 Codex 里叫「超高」,同一档两个名字容易让人以为是不同的东西);最低一档在各处统一叫「最低」。
- **「禁用图像生成」的说明文字重写**,不再让人误以为「要打开它 Codex 才正常」——**保持关闭即可**,打开反而会屏蔽生图端点。

## v0.5.8 - 2026-07-10

一版修复。**如果你用新版 Codex 且启动后一发消息就报错,这一版必装。**

### 修复

- **新版 Codex 一发消息就报 `Function 'image_gen.imagegen' conflicts with a hosted tool` 已修复**。此前 Quotio 给代理写的图像生成配置是「启用」这一档,而这一档会让代理**主动往每一笔 Codex 请求里塞一个内置的图像生成工具**;新版 Codex 自己也带了同名的图像工具,两个撞在一起,上游直接判定请求非法。现在改为「原样转发」——代理不再往你的请求里塞任何东西,冲突消失,而 `/v1/images/*` 生图端点**照常可用**(设置里的「禁用图像生成」开关打开时,才是真的全局禁用)。
- **改完自定义接口后,启动方案的模型下拉不再需要手动刷新**。此前在「服务商」页给自定义接口加了模型或设了模型前缀(如 `weiloo/gpt-5.5`),回到「智能体」页新建/编辑方案时,模型下拉还是旧的,得手动点一次刷新才能选到新模型。现在打开方案表单时会自动重新拉取。
- **编辑启动方案时不会再悄悄丢掉已选的模型**。如果方案里选的模型当时不在下拉列表中(代理没启动,或列表还没刷新到),下拉会显示空白,一按保存模型就被清掉了。现在已选的模型始终保留为选项。
- **Linux 下自动下载代理核心不再失败**。代理核心在 Linux 上以 `.tar.gz` 发布,而此前只会解 `.zip`,导致下载完装不上;现在支持 `.tar.gz` / `.tgz`,并会自动给解出来的二进制加上可执行权限。

## v0.5.7 - 2026-07-01

这一版给 Kiro **组织账号(AWS IAM Identity Center)**补齐了登录:既能在应用里一键登录,也修好了导入后用不了的问题。

- 修复**升级后本地设置被重置**:此前只要某个设置字段的格式跨版本变了(典型是枚举值改名),整份 `settings.json` 就会因解析失败被重置成默认、并在启动时覆盖原文件。现改为**容错解析**——逐字段抢救,只回退放不进当前版本的那个字段,其余(代理 / 端口 / 语言 / 主题 / 调度规则等)原样保留。
- 智能调度「临近刷新优先」不再死磕快耗尽的账号:选中账号任一窗口余量 ≤ 3% 时视为耗尽,**提前踢出代理池、改用满血的待命账号**,等它窗口刷新(余量回 > 5%)再放回(3%~5% 为迟滞缓冲,避免边界横跳)。此前余量只剩 3% 但 API 仍 `allowed` 的账号会被一直选中、请求全部 429,而满血的待命账号闲置不用——现已修复。

## v0.4.3 - 2026-06-20

### 新增

- 账号导入兼容 **sub2api** 导出格式(`type: "sub2api-data"`):自动展开 `accounts[]`,把每个 OpenAI 账号 `credentials` 里的令牌转换成 CLIProxyAPI codex 账号格式(`chatgpt_account_id`→`account_id`、`expires_at`→`expired`,丢弃 codex 不需要的字段),按 `codex-<邮箱>.json` 落地、导入即用;支持一个文件多账号。其它平台暂跳过。

### 修复

- 修复服务商页账号重复显示（ChatGPT Team 等）：真正的原因是**大小写**——代理的 `/auth-files` 会把账号文件名转成小写返回，而本地账号目录保留原始大小写；前端把「代理账号」与「本地补充账号」合并时按大小写敏感比对，导致文件名含大写字母的账号（如 `codex-MartilloOlivia…`）被当成「本地独有」又列了一遍（纯小写文件名的账号则正常）。改为大小写不敏感比对；后端账号信息富化、智能调度重启账号时的同名匹配也一并修正（避免「待命」账号被误显示为「已禁用」）。
- 升级时自动刷新内置组件：kiro-rs sidecar 与「按 key 路由」插件原先只在首次安装时落地、升级不会替换，旧机子可能一直用着旧的 sidecar/插件。现在改为与安装包内置版本按文件大小比对，不一致即覆盖管理目录的旧版本——与核心（CLIProxyAPI 一直是每次启动覆盖）保持一致，三个组件升级都会跟着更新。
